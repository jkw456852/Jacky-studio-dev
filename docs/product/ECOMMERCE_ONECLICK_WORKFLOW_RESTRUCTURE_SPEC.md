# 电商一键工作流重构方案

Last updated: 2026-03-28  
Owner: Codex  
Scope: Workspace 电商一键工作流阶段边界、状态机与 UI 语义重构

## 1. 背景

当前电商一键工作流已经具备完整链路能力，但随着补充编辑、批量生成、结果回看、模型锁定等能力不断叠加，阶段边界开始变得不清晰。

现在最突出的感受不是“功能不够”，而是：

- 用户以为自己进入了生图阶段，系统实际上还在做提示词定稿
- 流程进度在真正执行生成前就已经显示满格
- 模型锁定粒度过粗，无法覆盖不同组/不同任务的差异需求
- 失败恢复与部分成功策略不够产品化
- 单个阶段承担了过多职责，导致等待感和解释成本越来越高

这份方案的目标不是推翻现有实现，而是把当前可用链路重构成“阶段更清楚、等待更可解释、失败更好恢复、模型更可控”的版本。

## 2. 当前主要问题

### 2.1 阶段边界不清

当前 Step 7 名义上是“批量生成”，实际上同时承担：

- 提示词最终整理
- 生成模型执行
- 模型失败回退
- 结果持久化
- 结果排序与回写

这会造成明显的认知错位：用户点击的是“开始生成”，但前台体感却像“还在准备”。

### 2.2 进度语义失真

当前阶段进度里：

- `PLAN_SCHEMES` 文案仍然把提示词确认算进来
- `LOCK_MODEL` 已显示 `6/6`
- `BATCH_GENERATE` 在真正开始执行前，视觉上已经像完成阶段

这会削弱用户对流程的信任感。

### 2.3 模型控制粒度不合理

当前只有会话级 `selectedModelId`，这意味着：

- 同一轮所有方案组默认必须共用一个模型
- 局部例外只能通过重新锁整轮模型来绕开
- “全局映射 + 局部覆盖”的产品心智还没有落到状态结构上

### 2.4 失败恢复策略过硬

当前典型问题：

- 上传阶段只要部分失败，就中断整轮
- 生成阶段虽然支持失败重试，但更多是技术层按钮，不够像产品化恢复路径
- 用户需要自己理解哪些失败属于模型切换可恢复，哪些属于输入不足

### 2.5 单阶段职责过重

现在的批量阶段承担了过多状态变化和 UI 回写，后续继续叠加功能时，会越来越难维持：

- 文案一致性
- 状态可解释性
- 代码可维护性

## 3. 重构目标

### 3.1 产品目标

- 让每一步只承担一个主要任务
- 让“准备”和“执行”两个动作明确分开
- 让模型选择支持全局默认和局部覆盖并存
- 让失败恢复比现在更自然
- 让等待时间可以被解释，而不是只能靠安抚文案

### 3.2 技术目标

- 保留现有 store / controller / workflow_ui 架构
- 优先重排阶段语义和状态字段职责，不先做大规模推翻
- 允许分阶段迁移，避免一次性大改导致回归面过大

## 4. 目标流程

建议把当前流程从“6+1 展示步”调整为“7 个真实业务阶段”：

1. 上传商品
2. 商品分析
3. 补充关键信息
4. 参考图复核
5. 方案规划
6. 提示词定稿
7. 模型执行与结果筛选

说明：

- “提示词定稿”从当前 Step 7 中拆出来，成为独立阶段
- “模型锁定”不再独立作为一个完整业务阶段，而是挂靠在第 6、7 阶段中
- 第 7 阶段只负责执行、回退、结果落盘、筛选与补抽

## 5. 新阶段定义

### 阶段 1：上传商品

职责：

- 上传商品图
- 输入一句简要说明
- 建立当前会话

退出条件：

- 至少 1 张商品图可用

特殊策略：

- 允许部分上传失败后继续
- 失败文件单独列出，并提供补传入口

### 阶段 2：商品分析

职责：

- 输出商品定位、卖点和推荐图型

退出条件：

- 用户确认推荐图型组合

### 阶段 3：补充关键信息

职责：

- 收集真正影响方案质量的缺失信息

退出条件：

- 必填项达到最低可执行条件

### 阶段 4：参考图复核

职责：

- 判断哪些图适合做主体一致性和构图参考

退出条件：

- 用户确认参考图选择

### 阶段 5：方案规划

职责：

- 输出方案组
- 输出每组下的执行项
- 明确每条图位的目标、构图和比例

退出条件：

- 用户确认方案组与图位结构

不在本阶段做的事：

- 不做最终提示词整理
- 不执行生图

### 阶段 6：提示词定稿

职责：

- 为每个图位生成最终可编辑提示词
- 支持逐条改写
- 支持组内批量生成提示词
- 提供模型默认来源与局部覆盖入口

退出条件：

- 至少目标任务已具备可执行提示词

补充说明：

- 当前单独的“模型锁定”面板在这里并入
- 默认吃全局模型映射
- 可在组级或任务级覆盖

### 阶段 7：模型执行与结果筛选

职责：

- 按已定稿提示词执行生图
- 处理模型回退
- 回写结果
- 标记首选
- 补抽、删除、插画布、下载
- 同条历史结果对比

退出条件：

- 至少产出一批结果，用户可继续筛选或导出

不在本阶段做的事：

- 不再临时生成提示词
- 不再承担长解释型准备动作

## 6. 状态机调整建议

### 6.1 新 step 枚举建议

建议把现有：

- `WAIT_PRODUCT`
- `ANALYZE_PRODUCT`
- `SUPPLEMENT_INFO`
- `ANALYZE_IMAGES`
- `PLAN_SCHEMES`
- `LOCK_MODEL`
- `BATCH_GENERATE`
- `DONE`

调整为：

- `WAIT_PRODUCT`
- `ANALYZE_PRODUCT`
- `SUPPLEMENT_INFO`
- `ANALYZE_IMAGES`
- `PLAN_SCHEMES`
- `FINALIZE_PROMPTS`
- `EXECUTE_IMAGES`
- `DONE`

### 6.2 状态职责重分配

建议：

- `planGroups` 只负责方案结构与图位规划
- `batchJobs.finalPrompt` 的生成入口前移到 `FINALIZE_PROMPTS`
- `selectedModelId` 逐步拆成：
  - `defaultSelectedModelId`
  - `groupModelOverrides`
  - `jobModelOverrides`

执行优先级：

1. `jobModelOverrides`
2. `groupModelOverrides`
3. `defaultSelectedModelId`
4. 全局设置映射

### 6.3 执行阶段的 job 生命周期

建议让执行阶段的 job 状态更单纯：

- `idle`
- `queued`
- `generating`
- `done`
- `failed`

提示词相关状态从执行阶段中拿掉，不再混入 `imageStatus` 流程解释。

## 7. UI 结构调整建议

### 7.1 顶部步骤语义

- Step 5：方案规划
- Step 6：提示词定稿与模型确认
- Step 7：执行生图与筛选结果

这样可以显著降低“已经点了生图但还在准备”的违和感。

### 7.2 模型配置面板

建议把当前“模型锁定”区域变成：

- 默认模型来源
- 当前阶段默认模型
- 组级覆盖
- 任务级覆盖

UI 上不再使用“只有一个全局锁定”的表达方式。

### 7.3 执行阶段的总览卡

执行阶段总览只负责：

- 已完成数
- 排队数
- 失败数
- 本轮使用的默认模型
- 主操作按钮

不要再展示任何提示词准备信息。

### 7.4 提示词阶段的主内容

提示词阶段主内容建议是：

- 分组卡
- 组内图位卡
- 提示词编辑区
- 重写入口
- 模型覆盖入口

用户在这里完成“可执行性确认”。

### 7.5 执行阶段的主内容

执行阶段主内容建议是：

- 分组卡
- 图位执行卡
- 结果缩略图区
- 历史对比区

用户在这里完成“结果判断与筛选”。

## 8. 失败恢复策略建议

### 8.1 上传失败

改成：

- 成功图片先进入流程
- 失败图片列在“未成功上传”区域
- 用户可补传，不阻塞主流程

### 8.2 提示词失败

改成：

- 失败只影响当前图位
- 允许跳过失败项，继续执行已有提示词的项
- 提供“仅重跑失败提示词”入口

### 8.3 生图失败

改成：

- 失败原因分类展示：
  - 上游模型不可用
  - 请求超时
  - 输入不足
  - 结果持久化失败
- 提供：
  - 重试当前模型
  - 换模型重试
  - 跳过并继续下一项

## 9. 最小迁移路径

### Phase 1：语义校正

低风险，不改主数据结构：

- 修正步骤文案
- 修正进度语义
- 让 `PLAN_SCHEMES` 不再提提示词
- 让 `LOCK_MODEL/BATCH_GENERATE` 不再在执行前显示满格

### Phase 2：UI 拆层

中风险，尽量不动核心 skill 链：

- 新增 `FINALIZE_PROMPTS` 视图层
- 当前 Step 7 中的提示词编辑区迁移到新阶段
- 执行阶段只保留生成与结果筛选相关 UI

### Phase 3：状态拆分

中高风险：

- 拆分 `selectedModelId`
- 拆分提示词阶段状态与执行阶段状态
- 让 job 生命周期更纯粹

### Phase 4：恢复策略产品化

中高风险：

- 上传部分成功继续
- 明确失败分类
- 模型重试策略改为用户可理解的恢复路径

## 10. 本轮建议先落地的内容

建议先做下面 3 件，收益最大、风险最低：

1. 修正当前步骤文案与进度语义
2. 把提示词定稿从文案层先独立出来，避免继续误导用户
3. 在状态设计上为“组级/任务级模型覆盖”预留结构

## 11. 不建议立刻做的事

- 一次性重写全部 controller
- 一次性改 skill 调用链
- 一次性替换所有 workflow_ui message 类型

这些动作回归面太大，不适合直接硬切。

## 12. 判断标准

重构完成后，至少要满足：

- 用户不会再问“为什么还没开始生图就显示完成”
- 用户不会再把提示词准备误认为生图执行
- 用户能理解当前模型到底是全局默认还是局部覆盖
- 用户遇到失败时知道下一步该重试什么，而不是只能继续点按钮试错
- 第 7 步真正只剩执行和筛选，不再承担隐藏准备动作
