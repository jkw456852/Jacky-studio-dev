# 电商一键工作流交互优化交接单

Last updated: 2026-04-08 09:35 (UTC+8)  
Owner: XC 产品经理  
Target handoff: Architect / Coder

## 1. Task
- One-sentence task name:
  - 电商一键工作流步骤 5-7 交互减负与决策引导优化

## 2. Problem Statement
- Current behavior:
  - 当前电商一键工作流已经具备完整链路，但在步骤 5-7（方案规划、提示词定稿、批量执行）中，用户面对的是“高信息密度 + 多按钮并列 + 当前主决策不够清晰”的交互结构。
  - 抽屉顶部已经有步骤、进度、处理中说明，但还没有把“当前这一步用户唯一需要做的决定”明确提炼出来。
  - 方案规划区在成功路径下会一次性铺开较多分组、策略、镜头项、竞品提示和编辑字段；当真实任务量达到 8 组、32 个 job 时，浏览与聚焦成本明显升高。
  - 提示词定稿区与执行区虽然在视图层已开始拆分，但 CTA 仍有并列竞争，用户不容易判断系统推荐的下一步。
- Why it matters:
  - 这不是“功能缺失”问题，而是“成功路径理解成本”问题。
  - 如果用户在步骤 5-7 仍需要自己推断下一步，会导致确认慢、误操作多、回退多，削弱对自动工作流的信任。
- Evidence or user feedback:
  - 用户最新诉求是直接审视“电商一键工作流还有哪里可以优化改善的交互逻辑”。
  - 最新运行快照显示当前链路已能成功到达规划成功态，而非完全不可用：见 `.jk-studio-runtime/ecommerce-workflow-debug/latest-ecommerce-workflow-debug.json` 中 `stage = plan-groups-applied`、`planGroupCount = 8`、`batchJobCount = 32`。
  - 重构文档已明确指出目标应是“阶段更清楚、等待更可解释、失败更好恢复、模型更可控”。

## 3. Goal
- Primary goal:
  - 优化电商一键工作流步骤 5-7 的成功路径交互，让用户在每个阶段都清楚：
    1. 我现在在哪一步
    2. 我现在只需要做什么决定
    3. 系统建议我先看哪部分
- Success signal:
  - 用户首次进入方案规划时，不必滚完整个长列表才能开始确认。
  - 用户在提示词定稿阶段不会因多个并列按钮而犹豫“应该先整理提示词还是直接执行”。
  - 用户在执行阶段能优先看到失败 / 未完成 / 最近结果，而不是先陷入完整结果海。

## 4. Non-goals
- Out of scope:
  - 不重写整条 controller 或 skill 调用链。
  - 不在这一轮里一次性重构全部 workflow message 类型。
  - 不把 overlay 编辑器、结果深编辑区作为本轮主目标。
  - 不先处理业务正确性问题；本轮聚焦成功路径交互优化，而不是步骤三空结果 debug。

## 5. User Flow / Scope
- Entry point:
  - 用户从聊天区的电商工作流摘要卡进入大抽屉，主工作区为 `Workspace` 的电商一键工作流抽屉。
- Affected step(s):
  - 主要影响步骤：
    - Step 5：方案规划
    - Step 6：提示词定稿
    - Step 7：批量执行与结果筛选
- Key user path:
  1. 完成商品分析、补充信息和参考图分析
  2. 进入方案规划，查看并确认方案组
  3. 进入提示词定稿，补齐 / 修改提示词
  4. 进入执行阶段，运行任务并筛选结果

## 7. Key Findings
### 7.1 阶段语义已经拆开，但“阶段主决策”还没立起来
- 当前抽屉已有真实步骤顺序：
  - `WAIT_PRODUCT`
  - `ANALYZE_PRODUCT`
  - `SUPPLEMENT_INFO`
  - `ANALYZE_IMAGES`
  - `PLAN_SCHEMES`
  - `FINALIZE_PROMPTS`
  - `BATCH_GENERATE`
  - `DONE`
- 说明系统已经进入“规划 / 定稿 / 执行分离”方向，但顶部更多是状态说明，不是行动引导。

### 7.2 方案规划区信息层级过平
- 当前方案规划区同时展示：
  - 分组摘要
  - 分组策略
  - 竞品提示
  - 镜头项列表
  - 每个镜头项的编辑字段
  - AI 补图 / 手动新增 / 折叠等操作
- 在 8 组、32 job 的真实规模下，默认浏览成本偏高。

### 7.3 竞品信息当前更像增强项，而不是主路径核心项
- 最新成功快照里 `competitorAnalyses = []` 且 `competitorPlanningContext = null`，但规划仍成功。
- 这意味着竞品参考在主流程里应“按相关性浮现”，而不是长期占据主视觉注意力。

### 7.4 提示词定稿区 CTA 仍有竞争
- 当前会并列出现“进入工作台”“先批量整理提示词”“直接开始批量生成”等操作。
- 对熟练用户是灵活，对大多数用户则意味着决策负担。

### 7.5 执行区还缺少系统主动推荐视图
- 当前批量区已具备按组组织、结果对比等能力，但还不够像“任务驾驶舱”。
- 在大批量任务下，系统应该主动帮用户聚焦失败项、未执行项、最新结果，而不是默认完整展开。

## 8. Recommended Direction
### P0. 抽屉顶部增加“当前阶段唯一主决策”行动条
- 核心思路：把顶部从“说明型 UI”提升为“决策型 UI”。
- 每阶段仅保留：
  - 一个主动作
  - 一个次动作
  - 一个阻塞说明
- 示例：
  - 方案规划：主动作“确认方案结构”
  - 提示词定稿：主动作“补齐剩余提示词”
  - 执行阶段：主动作“执行未开始任务”

### P0. 方案规划区增加工作视图切换
- 建议在分组区顶部增加：
  - 全部
  - 待确认优先组
  - 仅问题组
- “问题组”建议先用轻量规则定义：
  - 低于镜头基线
  - 缺少策略字段
  - 当前无结果且优先级较高

### P0. 方案组与单图编辑分层
- 建议默认只展示：
  - 组目标摘要
  - 组策略
  - 图位清单摘要
- 点击某图位后，再打开单图编辑面板；不要默认全量铺开所有输入框。

### P1. 提示词定稿区 CTA 互斥化
- 建议按提示词就绪率切换主按钮：
  - 就绪率低：主按钮为“先批量整理提示词”
  - 就绪率高：主按钮为“开始执行已就绪任务”
- “进入工作台”降级为二级入口。

### P1. 竞品参考降噪
- 无竞品上下文时不展示竞品占位信息。
- 有竞品上下文时，仅在相关组 / 图位浮现单条提示。
- 更完整的竞品依据放入可展开证据层，而非主编辑层并排展示。

### P2. 执行区增加系统推荐聚焦模式
- 建议新增：
  - 优先看失败任务
  - 优先看未开始任务
  - 优先看最近结果
  - 优先看已设为首选的结果

## 9. Acceptance Criteria
1. 用户进入步骤 5-7 任一阶段后，能在抽屉顶部直接看到当前阶段唯一主动作。
2. 方案规划首次进入时，不需要先浏览全部分组才能开始操作。
3. 提示词定稿区的主按钮不再与其他同级按钮形成语义竞争。
4. 无竞品上下文时，主流程不出现高权重竞品提示。
5. 执行阶段支持快速切到失败 / 未开始 / 最新结果等高价值视图。

## 10. Suggested Implementation Order
### Phase 1：低风险 UI 收敛
1. 抽屉顶部增加“阶段主决策”行动条
2. 调整步骤 5-7 顶部 CTA 优先级
3. 补阶段内阻塞说明文案

### Phase 2：方案规划减负
1. 增加方案组视图切换
2. 增加“仅问题组 / 待确认组”派生筛选
3. 让单图编辑从默认展开改为按需展开

### Phase 3：执行区聚焦增强
1. 增加失败 / 未开始 / 最近结果聚焦模式
2. 优化执行区总览卡的统计优先级
3. 将“结果仓库”体验收敛为“任务驾驶舱”体验

## 11. Constraints
- Technical constraints:
  - 优先做 UI 层和本地派生状态，不先改 skill 链路。
  - 当前 `EcommerceOneClickCards` 文件体量较大，建议用局部派生和小范围抽离，避免大面积搬动导致回归。
- UX/copy constraints:
  - 文案必须强化“当前该做的唯一决定”，避免继续叠加解释型长文案。
- Time/risk constraints:
  - 先做低风险收益高的顶部行动条、CTA 收敛和视图筛选，不建议一步到位做全面信息架构重写。

## 12. Open Questions
- “问题组”的最小判定规则是否只做前端派生，还是需要 store 显式输出？
- 单图编辑是做右侧子面板，还是做组内抽屉展开？
- 提示词定稿阶段是否允许“部分就绪就执行”，以及默认推荐策略是保守还是激进？

## 13. Suggested First Engineering Scope
建议下一位优先收敛到以下实现范围：
1. 在抽屉顶部基于阶段状态补一层主决策行动条。
2. 在方案规划区为分组列表增加轻量筛选视图。
3. 在提示词定稿区根据 `promptReadyCount / total` 做 CTA 规则收敛。
4. 不改 skill 入参 / 出参，不改 controller 主链路，只做 UI 层增强与轻量派生。

## 14. Final Note
这轮最值得优先优化的，不是继续增加能力，而是把步骤 5-7 从“信息很多”整理成“当前只需做一个决定、只看最该看的那部分”。
