# 工程协作规范

## 1. 变更前流程（强制）
1. 高风险文件先备份。
2. 备份命名统一：`<原文件名>.bak-YYYYMMDD-HHMMSS`。
3. 先说明改动目标、影响范围，再开始编辑。
4. 开发前必须回看本次需求对应的产品方案、实施计划或专项文档原文，不允许凭记忆直接推进。
5. 回看后必须先确认：
   - 当前改动属于计划的哪一阶段
   - 当前代码已经做到哪一步
   - 本次改动是否超出该阶段边界
   - 本次实现是否存在概念跑偏
6. 若需求涉及智能体、编排、运行时、工具系统，必须额外检查：
   - 是否把“通用 agent”做偏成“planner 展示器”或“专用流程机”
   - 是否把内部能力错误外显成产品主流程
   - 是否重新引入隐藏规则兜底、静默降级、补丁套补丁
7. 每次进入新的具体实现步骤前，以及每完成一个子步骤后，都要做一次“跑偏复核”；发现偏离方案时，先停下来对齐，不继续顺手堆实现。

## 2. 编辑执行规范（强制）
1. 可按情况进行大面积重写；不强制小步分块。
2. 大面积重写前必须先备份目标文件。
3. 大面积重写后必须与备份对比回顾，重点检查逻辑、字段、样式与交互是否被误改。
4. 不覆盖已有改动：发现非本次改动内容，不擅自回滚。
5. 不做无关重构：除非明确需要，否则不顺手改 unrelated 代码。
6. 保持命名语义化：变量、函数、状态名可直接表达意图。
7. 涉及中文内容的文件，禁止使用 PowerShell 直接创建、重写或通过命令拼接方式编辑文件内容。
8. 涉及中文内容的文件，优先使用不会破坏 UTF-8 编码的编辑方式；若 `bash`/WSL 不可用，则必须先确认替代方案不会产生乱码，再执行修改。
9. 发现终端或脚本工具会污染中文编码时，立即停止该路径，不得继续试错式修改正式文件。
10. 编码敏感文件一旦出现乱码或工具兼容性差，优先小步补丁，不做整文件重写。
11. 改动模块时，必须同时检查是否还会跑回旧模块、旧组件、旧状态字段或旧链路；不能只看新模块本身是否能跑。
12. 若本次迭代已经存在新旧模块并行、旧入口残留、热更新可能命中旧片段、或旧状态仍能驱动旧 UI，必须在交付前明确排查并收口。
13. 旧模块、旧分支、旧兼容桥、旧常量若已无继续保留价值，能删就尽量删，不要长期挂在主链路旁边。
14. 若旧模块暂时不能删，必须同步说明：
15.   - 为什么现在不能删
16.   - 当前仍被哪些入口、数据或热更新链路引用
17.   - 准备如何隔离、下线或迁移
18.   - 在什么条件满足后可以彻底删除
19. 模块迭代后必须至少检查一次：
20.   - 新入口是否仍会打到旧实现
21.   - 旧变量/旧 UI 残片是否仍可能被引用
22.   - HMR/缓存恢复/持久化字段是否会把页面重新带回旧链路
23.   - 是否存在“源码已删，但运行态仍可能命中旧模块片段”的情况
24. 发现“新链路已接好，但运行时仍可能回到旧链路”时，不允许带着已知风险直接交付；要么当次收口删除，要么明确记录风险、原因和后续拆除方案。
25. 已验证可行的方法必须沉淀到项目准则中；同类问题下次优先复用已验证路径，不重复从零试错。
26. 若某种工具链或执行路径已经在当前仓库验证通过，应明确记录适用场景、限制条件与替代方案。
27. 当前仓库已验证可复用的方法包括：
28. `tsc --noEmit` 作为默认基础验证手段。
29. 中文/编码敏感文件避免 PowerShell 直接写入内容；优先使用安全补丁路径。
30. 当 `apply_patch.bat` 在 PowerShell 多行参数下不稳定时，可通过 `node` 调用底层 `codex.exe --codex-run-as-apply-patch` 传入 UTF-8 patch 内容。
31. 当 `bash`/WSL 不可用时，不继续沿该路径试错，立即切回当前仓库已验证的 Windows 本地安全方案。

## 3. 改后自检（强制）
1. 回看修改区：检查误删、错位、逻辑偏移、样式副作用。
2. 关键链路验证：至少执行一次类型检查（`tsc --noEmit`）。
3. 有条件时执行对应测试或最小功能自测。
4. 对照实施计划复核一次：本次产物是否真的推进了对应阶段目标，而不是引入了新的概念偏差。

## 4. 项目维护准则
1. 稳定性优先：先保证现有功能可用，再叠加新能力。
2. 单一职责：组件和模块职责清晰，降低耦合。
3. 向后兼容：尽量不破坏旧数据结构和旧调用路径。
4. 防御式编程：对空值、异常、边界值做兜底处理。
5. 可追踪性：每次改动明确“改了什么、为什么、影响哪里”。
6. 可回退性：保持每步都可定位、可撤回。
7. 新功能优先做干净主链路：这是新项目，不按“老项目打补丁”思路推进。
8. 禁止补丁套补丁式套娃：一个阶段能完成的事，不拆成多层补救链、拼接链、伪修复链。
9. 宁可暴露失败，也不伪装完成：不允许用兜底内容、拼接内容或伪结构假装功能已经做好。
10. 临时止血必须回收：排障时可短期加保护层，但确认方向后必须收回，不得沉淀成长期结构。
11. 一旦发现开始套娃，先停下来重整：优先重构主流程，不继续在坏结构上叠条件、叠分支、叠兜底。
12. 阶段边界要清楚：每个阶段只做本阶段的事，不让上游失败通过下游补丁偷偷扩散。
13. 方法资产化：本次排障、修复、验证中只要形成“已跑通路径”，都要回写到项目准则，供后续直接复用。
14. 已验证可行的方法必须沉淀进项目准则或对应专项文档；同类问题下次优先复用，不重复从零试错。
15. 新增“环境兼容性结论”时，至少记录：适用场景、已验证命令/路径、禁止路径、失败特征。
16. 模块进入重构/替换阶段后，默认目标不是“新旧长期并存”，而是“新链路稳定后尽快删除旧链路”。
17. 若因兼容旧数据、灰度迁移、跨页面共用、持久化恢复或运行时缓存等原因暂时必须保留旧模块，必须在对应文档或交付说明中写明删除条件与退出计划。
18. 对于本应由大模型或智能体负责的决策层，禁止规则兜底。
19. 决策层包括但不限于：任务编排、参考图角色分配、策略选择、智能路由、提示词规划、结果判定。
20. 若决策层依赖模型输出，则模型失败时必须显式报错或中止，不允许偷偷切回规则链路伪装成功。
21. 禁止“模型输出 + 隐藏规则主导”的伪智能实现；规则只能用于显式校验、开发诊断或非决策型预处理。

## 4.1 已验证方法记录（持续维护）
1. 文本/代码检索优先使用 `rg` / `rg --files`。
2. TypeScript 基线验证优先使用 `cmd /c node_modules\\.bin\\tsc --noEmit --pretty false`。
3. 文件修改优先使用补丁式编辑；能小步补丁时，不做整文件重写。
4. 中文或编码敏感文件修改，优先使用不会改写编码的补丁路径；禁止用 PowerShell 直接重写文件内容。
5. 在当前 Windows 环境下，如需通过命令行走补丁链路，优先让 `node` 读取 UTF-8 patch 内容后再调用底层补丁程序；不要把多行 patch 直接经 PowerShell 字符串或 `apply_patch.bat` 包装层传递。
6. 只读查看文件可使用 `Get-Content -LiteralPath`；但这不等于可安全用于中文文件写入。
7. 工作台图片生成/编辑链路中的“一致性检测”若已开启，首轮请求不要额外注入 `consistencyContext` 这类重约束；首轮只保留参考图与正常 prompt，一致性放到生成后复检，必要时再带修正提示自动重跑。
8. 当出现“关闭一致性检测不超时、开启后即使 1K 也超时”的现象时，优先排查首轮 prompt 注入和多模态约束膨胀，不要先怀疑参考图锚点本身。

## 5. UI 维护补充
1. 信息密度优先：减少无效留白，避免大面积空占位。
2. 层级清晰：分组、标题、标签、状态必须一眼区分。
3. 归属明确：列表项/卡片必须能快速判断属于哪个分组。
4. 保持一致：间距、字号、按钮风格遵循现有设计语言。

## 6. 沟通约定
1. 变更前：说明计划。
2. 变更中：同步进度和发现的问题。
3. 变更后：给出结果、验证情况、剩余风险与下一步建议。
